# servant-auth-roles

Role authorization for Servant.

Define a plain role enum, run one Template Haskell splice on it, and gate routes
with `RequireRole "tag" 'YourRole`. The combinator authenticates, decides
whether the user's role satisfies the requirement under a relation you pick, and
hands the handler a compile-time proof that the check passed. This proof can
then be used by downstream code to demand the privilege in its type signature
instead of re-checking it at runtime.

- **401** when the auth handler rejects the request (fatal — no alternatives tried).
- **403** when authentication succeeds but the role is insufficient (non-fatal —
  Servant falls through to the next `:<|>` alternative).

Three authorization schemes ship out of the box: hierarchical, exact match, and
permission sets.

## Quick start

```haskell
-- default-language: GHC2021
{-# LANGUAGE DataKinds #-}
{-# LANGUAGE OverloadedStrings #-}
{-# LANGUAGE TemplateHaskell #-}
{-# LANGUAGE TypeFamilies #-}
{-# LANGUAGE UndecidableInstances #-}
{-# OPTIONS_GHC -Wno-orphans #-}

module Main where

import Servant.Auth.Roles.TH
import Control.Monad.Except (throwError)
import Network.Wai (Request, requestHeaders)
import Network.Wai.Handler.Warp (run)
import Servant.API (Get, JSON, type (:<|>) (..), type (:>))
import Servant.API.Experimental.Auth (AuthProtect)
import Servant.Server (Context (..), Handler, Server, err401, serveWithContext)
import Servant.Server.Experimental.Auth (AuthHandler, AuthServerData, mkAuthHandler)

-- 1. A plain role enum. No deriving clause needed.
data UserRole = Viewer | Editor | Admin

-- 2. One splice. Constructor order defines the hierarchy: Admin > Editor > Viewer.
$(deriveOrdRole ''UserRole)

-- 3. Your auth type, indexed by the role.
newtype RoleAuth (r :: UserRole) = RoleAuth {rName :: String}

-- 4. Declare your @AuthServerData@ intance with the provided @SomeRole@ existential wrapper.
type instance AuthServerData (AuthProtect "role-auth") = SomeRole RoleAuth


-- 5. Define a standard @AuthHandler@, using @someUserRole@ to construct your @SomeRole@ value.
authHandler :: AuthHandler Request (SomeRole RoleAuth)
authHandler = mkAuthHandler $ \req ->
  case lookup "X-Role" (requestHeaders req) of
    Just "viewer" -> pure (someUserRole Viewer (RoleAuth "Reed"))
    Just "editor" -> pure (someUserRole Editor (RoleAuth "Lyxia"))
    Just "admin" -> pure (someUserRole Admin (RoleAuth "Sandy"))
    _ -> throwError err401

-- 6. Gate the routes.
type API =
       RequireRole "role-auth" 'Viewer :> "viewer" :> Get '[JSON] String
  :<|> RequireRole "role-auth" 'Admin  :> "admin"  :> Get '[JSON] String

-- A statically admin-gated subroutine. `IsAtleastAdmin` is generated by the
-- splice.
banUser :: (IsAtleastAdmin r) => RoleAuth r -> String
banUser auth = "banned by " <> rName auth

server :: Server API
server = viewerH :<|> adminH
  where
    viewerH :: Satisfies 'Viewer RoleAuth -> Handler String
    viewerH (Satisfies _ auth) = pure ("viewer: " <> rName auth)

    -- Matching on the proof constructor brings the `'Admin <= r` evidence into
    -- scope, which is what lets `banUser` be called here and nowhere else.
    adminH :: Satisfies 'Admin RoleAuth -> Handler String
    adminH (Satisfies UserRoleProof auth) = pure (banUser auth)

main :: IO ()
main = run 3000 $ serveWithContext (Proxy @API) (authHandler :. EmptyContext) server
```

```console
$ curl -H "X-Role: admin"  localhost:3000/admin   # "banned by Sandy"
$ curl -H "X-Role: editor" localhost:3000/admin   # 403 Forbidden: insufficient permissions
$ curl -H "X-Role: editor" localhost:3000/viewer  # "viewer: Lyxia"
$ curl                     localhost:3000/viewer  # 401
```

## The three schemes

Pick a deriver per role type. All three produce the same `RequireRole` interface.

| Deriver            | Sufficient when      | Actual role is  | Use for              |
|--------------------|----------------------|-----------------|----------------------|
| `deriveOrdRole`    | `required <= actual` | the role itself | ranked hierarchies   |
| `deriveEqRole`     | `required == actual` | the role itself | flat, unordered tags |
| `deriveMemberRole` | `required` ∈ actual  | a `[Role]` set  | permission sets      |

### Hierarchical `deriveOrdRole`

Constructor declaration order is the hierarchy; later constructors outrank
earlier ones.

```haskell
data UserRole = Viewer | Editor | Admin
$(deriveOrdRole ''UserRole)
```

`RequireRole "role-auth" 'Editor` admits `Editor` and `Admin`, rejects `Viewer`.

With hierarchical roles you can define overlapping routes indexed by the auth role:

```haskell
type API =
  RequireRole "role-auth" 'Admin  :> "dashboard"  :> Get '[JSON] String
    :<|> RequireRole "role-auth" 'Viewer :> "dashboard" :> Get '[JSON] String
```

A `'Viewer` will fail through the `'Admin` route but will satisfy the `'Viewer`
route. This allows you to have isolated handlers per auth roles for a single api
path.

#### Footguns

1. Routes are tried in declaration order so put the most restrictive role first.

2. Also note that each alternative re-runs the auth handler. Keep it in mind if
   your auth handler is expensive.

3. A public route alternative behind role gates is unaccessible. Authentication
   failure is fatal, so an unauthenticated request 401s at the first
   `RequireRole` and never reaches a trailing unauthenticated alternative on the
   same path.

### Exact match `deriveEqRole`

No ordering implied. Each role satisfies only itself.

```haskell
data Region = US | EU | APAC
$(deriveEqRole ''Region)

newtype RegionAuth (r :: Region) = RegionAuth {rTenant :: String}
type instance AuthServerData (AuthProtect "region-auth") = SomeRole RegionAuth

-- Only callable with proof that the tenant is in the EU.
gdprExport :: (IsEU r) => RegionAuth r -> String
gdprExport auth = "GDPR export for " <> rTenant auth

euH :: Satisfies 'EU RegionAuth -> Handler String
euH (Satisfies RegionProof auth) = pure (gdprExport auth)
```

A `US` tenant hitting `RequireRole "region-auth" 'EU` gets a 403.

### Permission sets `deriveMemberRole`

The auth value is indexed by the *set* of permissions held, and a requirement is
satisfied by membership.

```haskell
data Permission = PermRead | PermWrite | PermManage
$(deriveMemberRole ''Permission)

data PermAuth (ps :: [Permission]) = PermAuth

type instance AuthServerData (AuthProtect "perm-auth") = SomeRole PermAuth

authHandler :: AuthHandler Request (SomeRole PermAuth)
authHandler = mkAuthHandler $ \req ->
  case lookup "X-Perms" (requestHeaders req) of
    Just "reader" -> pure (somePermission [PermRead] PermAuth)
    Just "editor" -> pure (somePermission [PermRead, PermWrite] PermAuth)
    _ -> throwError err401
```

A gate can require a single permission, or a list, in which case every
element must be held (subset):

```haskell
type PermAPI =
       RequireRole "perm-auth" 'PermWrite            :> "write"     :> Get '[JSON] String
  :<|> RequireRole "perm-auth" '[PermRead, PermWrite] :> "readwrite" :> Get '[JSON] String
```

Because membership over an abstract set is not something the constraint solver
can decide, the proof is turned into a constraint explicitly, with `withMember`
(single permission) or `withAllMembers` (a required list):

```haskell
-- Demands the membership constraint, not a proof value.
writeUser :: (HasPermWrite ps) => PermAuth ps -> IO ()
writeUser PermAuth = pure ()

writeH :: Satisfies 'PermWrite PermAuth -> Handler String
writeH (Satisfies proof auth) =
  liftIO (withMember proof (writeUser auth)) >> pure "write ok"

-- The subset proof releases *both* memberships at once.
readWriteH :: Satisfies '[PermRead, PermWrite] PermAuth -> Handler String
readWriteH (Satisfies proof auth) =
  liftIO (withAllMembers proof (writeUser auth)) >> pure "read+write ok"
```

## What the splices generate

For a role type `T` with constructor `C`:

| Name                             | From                            | What it is                                                 |
|----------------------------------|---------------------------------|------------------------------------------------------------|
| singletons for `T`               | all three                       | via `genSingletons`, plus `SEq`/`SOrd`/`SDecide` as needed |
| `TProof`                         | `deriveOrdRole`, `deriveEqRole` | the proof constructor to match on                          |
| `THere`, `TThere`                | `deriveMemberRole`              | membership witnesses                                       |
| `TAllNil`, `TAllCons`            | `deriveMemberRole`              | subset witnesses for a required list                       |
| `someT`                          | all three                       | packs a runtime role into `SomeRole`                       |
| `IsAtleastC r`                   | `deriveOrdRole`                 | constraint alias for `('C <= r) ~ 'True`                   |
| `IsC r`                          | `deriveEqRole`                  | constraint alias for `r ~ 'C`                              |
| `HasC ps`                        | `deriveMemberRole`              | constraint alias for `Member 'C ps`                        |
| `ActualK`, `Decidable` instances | all three                       | the relation itself                                        |

The constraint aliases are the point of contact for your own code. Write
`IsAtleastAdmin r =>` on a function and it becomes uncallable without a gate
having proved it.

Import `Servant.Auth.Roles.TH` and you get the combinator, the classes, the
derivers, and the singletons Prelude the generated code needs.

## Mixing with `AuthProtect`

`RequireRole` and `AuthProtect` share one context entry and one `AuthServerData`
instance, so they mix freely:

```haskell
type API =
       RequireRole "role-auth" 'Admin :> "admin"   :> Get '[JSON] String
  :<|> AuthProtect "role-auth"        :> "profile" :> Get '[JSON] String
  :<|> "public" :> Get '[JSON] String
```

The `AuthProtect` handler receives the raw `SomeRole RoleAuth` authenticated,
but with no proof attached, so you unpack the existential yourself:

```haskell
profileH :: SomeRole RoleAuth -> Handler String
profileH (SomeRole _ auth) = pure (rName auth)
```

No extra context entries are needed for either.

## Without Template Haskell

`Servant.Auth.Roles` is the low-level interface; the derivers are a convenience
over it. To write a relation by hand, supply singletons for the role type, an
`ActualK` instance (the kind the requirement is checked against), a `Proof` data
instance, and a `Decidable` instance:

```haskell
type instance ActualK (req :: UserRole) = UserRole

data instance Proof (req :: UserRole) (r :: UserRole) where
  UserRoleProof :: (req <= r) ~ 'True => Proof req r

instance Decidable (req :: UserRole) where
  decideRole a b = case a %<= b of
    STrue -> Just UserRoleProof
    SFalse -> Nothing
```

Required and actual kinds may differ, which is exactly how the permission-set
scheme fits: a required `'Permission` is checked against an actual
`[Permission]`.
